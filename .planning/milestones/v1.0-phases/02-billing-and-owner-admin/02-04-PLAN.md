    1. Update apps/owner/src/app/api/auth/login/route.ts:
       - After password verification, check ownerUser.totpEnabled
       - If totpEnabled is true:
         - Generate a short-lived temp token (5 minutes, type: 'totp-pending') using signOwnerToken pattern but with different type
         - Return { requiresTotp: true, tempToken } — do NOT return access/refresh tokens yet
       - If totpEnabled is false:
         - Existing behavior: return { accessToken, refreshToken }

    2. Create apps/owner/src/app/api/auth/totp-verify/route.ts:
       - POST handler accepting { tempToken: string, totpCode: string }
       - Verify tempToken (must be type: 'totp-pending', not expired)
       - Extract ownerUserId from tempToken, load OwnerUser, get totpSecret
       - Call verifyTotp(totpSecret, totpCode)
       - If valid: issue real accessToken + refreshToken via signOwnerToken
       - If invalid: return 401 { error: 'Invalid TOTP code' }

    3. Create apps/owner/src/app/api/auth/totp-setup/route.ts:
       - POST handler (requires auth — owner must be logged in)
       - Body: { action: 'generate' } or { action: 'enable', totpCode: string }
       - generate: call generateTotpSecret(email), save totpSecret to OwnerUser (but don't enable yet), return { otpauthUrl, qrCode }
       - enable: verify the TOTP code against stored secret, if valid set totpEnabled = true, return { success: true }
       - This two-step prevents enabling TOTP without confirming the authenticator app works

    4. Update apps/owner/src/middleware.ts:
       - Add IP allowlist check BEFORE JWT verification
       - Read OWNER_ADMIN_IP_ALLOWLIST env var (comma-separated IPs/CIDRs, e.g., "10.0.0.0/8,192.168.1.0/24,127.0.0.1")
       - If env var is set and non-empty: check request IP against allowlist
       - If IP not in allowlist: return 403 response
       - If env var is not set or empty: skip IP check (development mode)
       - Use simple IP match (exact match + CIDR /8, /16, /24 support)
       - The middleware.ts runs in Edge runtime — cannot use Node.js net module. Implement simple CIDR matching with bitwise operations on parsed IP octets.

    5. Create apps/api/src/middleware/impersonation-guard.ts:
       - Export async function blockImpersonationWrites(request: FastifyRequest, reply: FastifyReply)
       - Check request.user for readOnly === true or impersonatedBy being set
       - If either: return 403 { error: 'READ_ONLY_SESSION', message: 'Impersonation sessions are read-only. Exit impersonation to make changes.' }
       - This is a Fastify preHandler

    6. Register impersonation guard in apps/api/src/server.ts:
       - Add blockImpersonationWrites as preHandler on the protected scope
       - It must run AFTER authPreHandler (which sets request.user)
       - The main API verifies impersonation tokens using IMPERSONATION_JWT_SECRET
       - Update authPreHandler or add a secondary check: if JWT contains 'impersonatedBy' claim, it was signed with IMPERSONATION_JWT_SECRET. The auth middleware must try verifying with IMPERSONATION_JWT_SECRET as a fallback when standard JWT_SECRET verification fails.

    7. Create impersonation-guard.test.ts:
       - Test: POST request with readOnly: true user returns 403
       - Test: PUT request with impersonatedBy set returns 403
       - Test: GET request with readOnly: true passes through (no block)
       - Test: POST request without readOnly passes through
