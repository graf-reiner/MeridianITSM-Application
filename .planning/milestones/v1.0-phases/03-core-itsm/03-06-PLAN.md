1. **Create `apps/api/src/routes/v1/settings/users.ts`** (SETT-01):
   - GET `/api/v1/settings/users` — List users (paginated, searchable by name/email, filterable by isActive/systemRole)
   - GET `/api/v1/settings/users/:id` — Get user detail (include roles, groups)
   - POST `/api/v1/settings/users` — Create user: `{ email, firstName, lastName, password, systemRole?, customRoleId? }`. Hash password with `@node-rs/bcrypt` (`hash(password, 12)`). Validate email unique per tenant.
   - PATCH `/api/v1/settings/users/:id` — Update user: `{ firstName?, lastName?, isActive?, systemRole?, customRoleId? }`. Never allow password update through this route.
   - POST `/api/v1/settings/users/:id/reset-password` — Admin password reset: `{ newPassword }`. Hash and update passwordHash. Force: `passwordResetRequired = true` (if field exists) or store in customFields.
   - POST `/api/v1/settings/users/:id/disable` — Set `isActive = false`
   - POST `/api/v1/settings/users/:id/enable` — Set `isActive = true`

2. **Create `apps/api/src/routes/v1/settings/roles.ts`** (SETT-02):
   - GET `/api/v1/settings/roles` — List all roles (system + custom)
   - POST `/api/v1/settings/roles` — Create custom role: `{ name, permissions: string[] }`. Validate permissions against known permission list. Set `isSystem = false`.
   - PATCH `/api/v1/settings/roles/:id` — Update role (block editing system roles: check `isSystem === false`)
   - DELETE `/api/v1/settings/roles/:id` — Delete custom role (block if isSystem=true, block if users assigned)

3. **Create `apps/api/src/routes/v1/settings/groups.ts`** (SETT-03):
   - GET `/api/v1/settings/groups` — List groups with member count
   - POST `/api/v1/settings/groups` — Create group: `{ name, email?, description? }`
   - PATCH `/api/v1/settings/groups/:id` — Update group
   - DELETE `/api/v1/settings/groups/:id` — Delete group (cascade deletes UserGroupMember)
   - POST `/api/v1/settings/groups/:id/members` — Add member: `{ userId, role? }`
   - DELETE `/api/v1/settings/groups/:id/members/:userId` — Remove member
   - GET `/api/v1/settings/groups/:id/members` — List members with user details

4. **Create `apps/api/src/routes/v1/settings/queues.ts`** (SETT-04):
   - GET `/api/v1/settings/queues` — List queues
   - POST `/api/v1/settings/queues` — Create: `{ name, autoAssign?, defaultAssigneeId?, assignmentRules? }`
   - PATCH `/api/v1/settings/queues/:id` — Update
   - DELETE `/api/v1/settings/queues/:id` — Delete (block if tickets assigned)

5. **Create `apps/api/src/routes/v1/settings/categories.ts`** (SETT-06):
   - GET `/api/v1/settings/categories` — List categories as flat list with parentId for client-side tree building. Include `childCount` via `_count`.
   - GET `/api/v1/settings/categories/tree` — List as hierarchical tree (recursive query or application-level nesting)

   - PATCH `/api/v1/settings/categories/:id` — Update (validate no circular parentId reference)
   - DELETE `/api/v1/settings/categories/:id` — Delete (block if children exist or tickets use it)
