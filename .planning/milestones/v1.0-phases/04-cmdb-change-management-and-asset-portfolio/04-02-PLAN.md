1. createCI(prisma, tenantId, data, userId):
   - Use $transaction with $queryRaw FOR UPDATE lock for sequential ciNumber:
     SELECT COALESCE(MAX("ciNumber"), 0) + 1 AS next FROM cmdb_configuration_items WHERE "tenantId" = ${tenantId}::uuid FOR UPDATE
   - Create CmdbConfigurationItem with: name, type (default OTHER), status (default ACTIVE), environment (default PRODUCTION), categoryId, assetId, agentId, ownerId, siteId, attributesJson
   - Log creation in CmdbChangeRecord: { tenantId, ciId, changeType: 'CREATED', changedBy: 'USER', userId }
   - Return created CI with ciNumber

2. getCI(prisma, tenantId, ciId):
   - Find by id + tenantId, include: category, sourceRels (with target CI name/type), targetRels (with source CI name/type), changeRecords (ordered by createdAt DESC, limit 50), ticketLinks (with ticket title/ticketNumber)

3. listCIs(prisma, tenantId, filters):
   - Filters: type?, status?, environment?, categoryId?, search? (matches name, ciNumber cast to text)
   - Pagination: page (default 1), pageSize (default 25)
   - Return { data: CI[], total: number, page: number, pageSize: number }

4. updateCI(prisma, tenantId, ciId, data, userId):
   - In transaction: fetch current CI, compare changed fields, create CmdbChangeRecord for EACH changed field with oldValue/newValue
   - changedBy: 'USER', userId set
   - Update the CI

5. deleteCI(prisma, tenantId, ciId, userId):
   - Delete related CmdbRelationships (source and target), CmdbTicketLinks, CmdbChangeRecords
   - Delete the CI
   - Or soft-delete by setting status to DECOMMISSIONED (preferred)

6. createRelationship(prisma, tenantId, data):
   - data: { sourceId, targetId, relationshipType, description? }
   - Validate both CIs exist and belong to tenantId
   - Prevent self-referencing (sourceId !== targetId)
   - Create CmdbRelationship. Unique constraint [sourceId, targetId, relationshipType] prevents duplicates

7. deleteRelationship(prisma, tenantId, relationshipId):
   - Delete by id + tenantId

8. getImpactAnalysis(prisma, tenantId, rootCiId, maxDepth = 2):
   - Use PostgreSQL recursive CTE (from RESEARCH.md Pattern 3):
     WITH RECURSIVE impact_graph AS (
       SELECT r."targetId" AS "ciId", 1 AS depth, r."relationshipType", ARRAY[${rootCiId}::uuid, r."targetId"] AS path
       FROM cmdb_relationships r
       WHERE r."sourceId" = ${rootCiId}::uuid AND r."tenantId" = ${tenantId}::uuid
       UNION ALL
       SELECT r."targetId", ig.depth + 1, r."relationshipType", ig.path || r."targetId"
       FROM cmdb_relationships r
       INNER JOIN impact_graph ig ON r."sourceId" = ig."ciId"
       WHERE r."tenantId" = ${tenantId}::uuid AND ig.depth < ${maxDepth} AND NOT (r."targetId" = ANY(ig.path))
     )
     SELECT DISTINCT ON ("ciId") * FROM impact_graph ORDER BY "ciId", depth LIMIT 10000
   - Also traverse reverse direction (targetId -> sourceId) for upstream analysis
   - Join result CIs to get name, type, status for each impacted CI
   - Return { rootCi: CI, impacted: ImpactedCI[], totalCount: number }
   - maxDepth capped at 5 (per RESEARCH.md)

9. listCIChangeHistory(prisma, tenantId, ciId, page = 1, pageSize = 50):
   - CmdbChangeRecord by ciId + tenantId, ordered by createdAt DESC
   - Return paginated result

10. createCategory(prisma, tenantId, data):
    - data: { name, slug, icon?, color?, parentId?, description? }
    - If parentId provided, validate it exists and belongs to tenantId
    - Cycle detection via raw SQL (same pattern as Phase 3 category):
      WITH RECURSIVE ancestors AS (
        SELECT id, "parentId" FROM cmdb_categories WHERE id = ${parentId}::uuid AND "tenantId" = ${tenantId}::uuid
        UNION ALL
        SELECT c.id, c."parentId" FROM cmdb_categories c JOIN ancestors a ON c.id = a."parentId"
      )
      SELECT id FROM ancestors WHERE id = ${newCategoryId}::uuid
    - If cycle detected, throw Error('Category hierarchy cycle detected')

11. listCategories(prisma, tenantId):
    - Return all categories with children relation for tree building
