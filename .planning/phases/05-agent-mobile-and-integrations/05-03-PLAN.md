2. **Push notification worker** — Create `apps/worker/src/workers/push-notification.ts`:

   ```typescript
   import { Worker, type Job } from 'bullmq';
   import { Expo, type ExpoPushMessage } from 'expo-server-sdk';
   import { prisma } from '@meridian/db';
   import { bullmqConnection } from '../queues/connection.js';
   import { QUEUE_NAMES } from '../queues/definitions.js';

   const expo = new Expo();

   export interface PushJobData {
     tenantId: string;
     userId: string;
     notificationType: string;  // NotificationType enum value
     title: string;
     body?: string;
     screen?: string;   // 'ticket', 'change', 'cab', etc.
     entityId?: string;
   }
   ```

   Worker logic:
   - Fetch user record to check pushPreferences: `prisma.user.findFirst({ where: { id: userId, tenantId }, select: { pushPreferences: true } })`
   - Check preferences: if `pushPreferences` is not null and `pushPreferences[notificationType] === false`, skip this push (return early)
   - Default: null pushPreferences means all types enabled
   - Fetch active DeviceTokens: `prisma.deviceToken.findMany({ where: { tenantId, userId, isActive: true } })`
   - If no tokens, return
   - Filter tokens to valid Expo push tokens: `Expo.isExpoPushToken(t.token)`
   - Build ExpoPushMessage[] with: `{ to: t.token, title, body, data: { screen, entityId }, sound: 'default' }`
   - Send via `expo.chunkPushNotifications(messages)` and `expo.sendPushNotificationsAsync(chunk)`
   - Handle DeviceNotRegistered errors: for each ticket with `status === 'error'` and `details?.error === 'DeviceNotRegistered'`, mark the corresponding DeviceToken as `isActive: false`
   - Worker options: `{ connection: bullmqConnection, concurrency: 3 }`

1. **Alert channel routes** — Create `apps/api/src/routes/v1/settings/alerts.ts` exporting `alertChannelRoutes(app: FastifyInstance)`:

   **POST /api/v1/settings/alerts** (JWT, admin — hasPermission 'settings.update'):
   - Accept body: `{ name: string, channelType: 'EMAIL' | 'SLACK' | 'TEAMS', config: Record<string, unknown>, events: string[] }`
   - Config validation per channel type:
     - EMAIL: `{ recipients: string[] }` — array of email addresses
     - SLACK: `{ webhookUrl: string }` — Slack incoming webhook URL (must start with `https://hooks.slack.com/`)
     - TEAMS: `{ webhookUrl: string }` — Teams connector URL (must start with `https://`)
   - Create AlertConfiguration: `{ tenantId, name, channelType, config, events, isActive: true }`
   - Return 201 with the created alert channel

   **GET /api/v1/settings/alerts** (JWT, admin):
   - List all alert configurations for tenant
   - Return `{ id, name, channelType, events, isActive, createdAt }`
   - Do NOT return config (may contain secrets like webhook URLs) in list view

   **GET /api/v1/settings/alerts/:id** (JWT, admin):
   - Return full alert configuration including config

   **PATCH /api/v1/settings/alerts/:id** (JWT, admin):
   - Accept partial body: `{ name?, config?, events?, isActive? }`
   - Update and return

   **DELETE /api/v1/settings/alerts/:id** (JWT, admin):
   - Delete AlertConfiguration where id + tenantId
   - Return 200 `{ ok: true }`

   **POST /api/v1/settings/alerts/:id/test** (JWT, admin):

   - Return 200 `{ success: true }` or `{ success: false, error: string }`
